# 競馬予想アプリ プロジェクト概要

> この文書はPhase0で作成したプロジェクト概要です。現在の実装状態、実装済み機能、未実装機能は [PROJECT_STATUS.md](./PROJECT_STATUS.md) を正とします。

## 1. 文書の目的

本書は、競馬予想アプリの目的、対象範囲、開発原則、および成功条件を定義する。

詳細な開発フェーズは [PHASE_PLAN.md](./PHASE_PLAN.md)、技術構成は [ARCHITECTURE.md](./ARCHITECTURE.md)、データベース設計は [DATABASE_DESIGN.md](./DATABASE_DESIGN.md)、Codexを含む開発時の規約は [CODEX_RULES.md](./CODEX_RULES.md)を参照する。

## 2. プロダクトの目的

合法的に取得した競馬データを整理・可視化し、利用者がレースや出走馬を比較検討できるWebアプリを提供する。

初期段階では、取得済みの事実データを正確かつ分かりやすく表示することに集中する。AIによる予測や高度な分析は、十分なデータ品質と検証基盤を整えた後のフェーズで追加する。

## 3. 基本方針

- 個人開発として、運用負荷と固定費を抑える。
- Webアプリを先行し、将来のモバイル対応を妨げない設計にする。
- データ取得は、契約、利用許諾、提供仕様を順守した手段だけを使用する。
- スクレイピングは行わない。
- Phase 1では、データとして合法的に取得できる情報だけを扱う。
- 表示データには、可能な限り出典、更新時刻、確定・速報などの状態を持たせる。
- 的中、払戻、利益を保証する表現は使用しない。
- 予想や分析を提供する将来フェーズでも、娯楽・情報提供サービスとして誤認を招かない表現を徹底する。
- 将来のAI・機械学習に備え、データが「いつ利用可能だったか」を記録する。
- Supabase固有機能への依存範囲を管理し、AWS RDS PostgreSQLへ移行可能な構造を保つ。

## 5. Phase 1の対象

### 対象に含めるもの

- ユーザー認証
- 開催、競馬場、レースの参照
- 出走馬、枠番、馬番、騎手、調教師など取得可能な基本情報の参照
- 合法的に取得できる過去成績、レース結果、払戻などの参照
- 利用許諾の範囲内で取得できるオッズ情報の参照
- JRA-VAN、JRDB、許諾済みCSVなどからのデータ取込を想定した受け口
- データ取込履歴、エラー、出典、時刻の管理
- お気に入りやメモなど、事実データを変更しないユーザー固有機能
- 基本的な運用監視、監査ログ、バックアップ方針

実際に表示・保存できる項目は、各データ提供元との契約および利用条件によって確定する。

### 対象に含めないもの

- AI予測
- 機械学習モデルの学習・推論
- 特徴量生成
- ペース予想
- 位置取り予想
- 自動投票、馬券購入連携
- 的中保証、利益保証、回収率保証
- スクレイピング
- 契約または利用許諾が確認できないデータの取込・再配布

## 6. 想定するデータ取得元

- JRA-VAN
- JRDB
- 利用許諾が明確なCSVファイル
- 将来契約する公式APIまたは正規データフィード

データ提供元ごとに、以下を事前に確認する。

- 取得方法が利用規約に適合していること
- 保存可能なデータと保存期間
- Web上で表示可能な項目
- 加工、分析、再配布、商用利用の可否
- 出典表示義務
- 契約終了時の削除義務
- 更新頻度、取得上限、障害時の扱い

技術的に取得できることは、利用・表示・再配布が許可されていることを意味しない。

## 7. 対象ユーザー

Phase 1では、次の利用者を主な対象とする。

- レース情報を一覧で確認したい利用者
- 出走馬や過去成績を比較したい利用者
- 自分のお気に入り、注目馬、メモを管理したい利用者
- 将来の予測機能に先立ち、整理された競馬データを利用したい利用者

## 8. 非機能要件

### セキュリティ

- 認証はSupabase Authを利用する。
- 認可はサーバー側で検証し、画面上の表示制御だけに依存しない。
- 管理用キー、データ提供元の認証情報、DB接続情報をクライアントへ公開しない。
- 個人情報と競馬データを論理的に分離する。
- Supabase Row Level Securityを使用する場合も、ポリシーを明示的にテストする。

### データ品質

- 外部IDと内部IDを分離する。
- 同一データの再取込で重複しない、冪等な取込を基本とする。
- 速報値、確定値、訂正値を区別する。
- 取込元、取込バッチ、原データとの対応を追跡可能にする。
- `available_at`、`observed_at`、`imported_at`を用途に応じて保存する。

### 可用性・性能

- Phase 1では、個人開発で維持可能な範囲の可用性を目標とする。
- 一覧検索に必要なインデックスを設け、過剰な先行最適化はしない。
- 大量になりやすいオッズ履歴や取込ログは、主要マスタ・レース情報から分離する。

### 移植性

- PostgreSQL標準機能を優先する。
- DBスキーマ変更はマイグレーションで管理する。
- アプリケーションの主要なDBアクセスはDrizzle ORM経由とする。
- Supabase AuthのユーザーIDを業務テーブルへ直接拡散させない。
- Supabase固有のEdge FunctionsやRealtimeは、明確な必要性が生じるまで中核処理に使用しない。

## 9. Phase 1の完了条件

- 認証済みユーザーが、許諾された範囲の開催・レース・出走情報を閲覧できる。
- 許諾されたデータファイルまたはデータフィードを安全に取り込める。
- 同じデータを再取込しても、意図しない重複が発生しない。
- データの出典と取込時刻を追跡できる。
- 速報・確定・訂正を区別できる。
- ユーザー固有データを他ユーザーが閲覧・更新できない。
- CIで型チェック、Lint、テスト、マイグレーション整合性を確認できる。
- バックアップとリストアの手順が文書化されている。
- 禁止事項に該当する機能・表現・データ取得が含まれていない。

## 10. 成功指標

Phase 1では予想精度や利益ではなく、次を重視する。

- データ取込成功率
- データ更新の遅延
- 重複・欠損・不整合の件数
- ページ表示速度とエラー率
- 継続利用率
- お気に入り、メモなどの利用状況
- 問い合わせや訂正依頼への追跡可能性

## 11. 重要な免責方針

本サービスは競馬に関する情報提供を目的とし、的中や利益を保証しない。将来予測機能を追加する場合も、予測は不確実であり、最終判断は利用者自身が行うことを明示する。

法令、データ提供契約、表示義務、商用利用条件については、公開・収益化前に最新条件を確認し、必要に応じて専門家へ相談する。
