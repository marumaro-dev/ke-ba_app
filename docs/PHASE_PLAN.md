# 開発フェーズ計画

> この文書は初期フェーズ計画を現在の進捗に合わせて更新したものです。実装済み機能の詳細は [PROJECT_STATUS.md](./PROJECT_STATUS.md) を参照してください。

## 1. 文書の目的

本書は、競馬予想アプリを小さく安全に立ち上げ、データ品質を確保してから分析・AI機能へ進むための段階的な計画を定義する。

フェーズ番号は開発順序を示す。後続フェーズの機能を先行実装する場合は、目的、依存関係、データ利用条件を文書化したうえで合意を得る。

## 2. 共通ゲート

すべてのフェーズで、次の条件を満たす。

- スクレイピングを使用しない。
- 契約・利用許諾が確認できるデータだけを扱う。
- 的中保証、利益保証に当たる機能や表現を使用しない。
- 秘密情報をリポジトリやクライアントへ含めない。
- DB変更をマイグレーションで管理する。
- `available_at`、`observed_at`、`imported_at`を考慮する。
- テストとCIを通過してから変更を統合する。
- Supabase PostgreSQLから標準PostgreSQLへ移せる設計を維持する。

- セキュリティ、バックアップ、監視、障害対応の基本方針を決定する。
- 初期コスト上限とサービスの料金アラートを決定する。

### 成果物

- `PROJECT_OVERVIEW.md`
- `PHASE_PLAN.md`
- `ARCHITECTURE.md`
- `DATABASE_DESIGN.md`
- `CODEX_RULES.md`
- データ利用条件一覧

### 完了条件

- 未確定事項が課題として可視化されている。

## 4. Phase 1: 事実データ参照MVP

### 目的

合法的に取得できる事実データを、正確かつ安全に取り込み、Web上で参照できる最小プロダクトを公開する。

### 対象機能

#### 基盤

- Next.js、TypeScript、Drizzle ORM、Zodの初期設定
- Supabaseの開発・本番プロジェクト設定
- GitHub ActionsによるCI
- 環境変数とシークレットの管理
- エラー監視、構造化ログ、ヘルスチェック

#### 認証・ユーザー機能

- サインアップ、ログイン、ログアウト
- パスワード再設定などSupabase Authの基本フロー
- ユーザープロフィール
- お気に入り、メモ
- ユーザー単位の認可

#### データ取込

- JRA-VAN、JRDB、CSVなど、利用許諾済みの入力方式に対応する取込境界
- ファイル形式・項目のZod検証
- 取込バッチの記録
- ステージング領域への保存と業務テーブルへの反映
- 冪等なUPSERT
- エラー行の隔離と再処理
- 出典、外部ID、各種時刻の保存
- 速報、確定、訂正状態の管理

#### データ参照

- 開催日・競馬場別のレース一覧
- レース詳細
- 出走馬、騎手、調教師などの基本情報
- 利用許諾範囲内の過去成績、結果、払戻、オッズ
- データ更新日時、状態、出典の表示
- 基本的な検索、絞り込み、並び替え

### 明確な対象外

- AI予測
- 特徴量生成
- ペース予想
- 位置取り予想
- 独自指数
- 自動投票
- 収益・的中の保証
- 許諾されていないデータの取得・表示

### 推奨スプリント順

1. 開発基盤、CI、環境分離
2. DBマイグレーションと主要マスタ
3. データ提供元・取込バッチ・ステージング
4. 最小のCSV取込と冪等性
5. レース・出走・結果の参照API
6. 一覧・詳細画面
7. Supabase Authとユーザー固有機能
8. 監視、バックアップ、セキュリティ確認
9. 利用規約、プライバシーポリシー、免責表示
10. 本番公開

### 完了条件

- [PROJECT_OVERVIEW.md](./PROJECT_OVERVIEW.md)のPhase 1完了条件をすべて満たす。
- データ提供元ごとの契約条件と実装範囲が対応付けられている。
- 取込失敗時に原因を特定し、再処理できる。
- 時点整合性を確認するテストがある。
- RLSまたはサーバー認可のテストがある。
- バックアップからの復元手順を試行済みである。

## 5. Phase 2: 特徴量生成

### 目的

Phase1で取り込んだ事実データから、予測スコアや将来の学習データに使える特徴量を、データリークを避けて生成・保存する。

### 実装済み機能

- `feature_definitions`
- `feature_snapshots`
- `feature_generation_batches`
- P0特徴量
- P1特徴量
- P2特徴量
- 特徴量生成CLI
- 特徴量生成履歴画面
- レース詳細での特徴量表示

### 注意点

- 対象レース自身の結果を特徴量に使わない。
- `available_at <= as_of_at` の情報だけを使う。
- P3脚質、ペース予想、位置取り予想は未実装。

### 完了条件

- P0/P1/P2特徴量を生成・保存できる。
- 同じ条件で複数回実行しても重複しない。
- レース詳細と生成履歴画面で特徴量を確認できる。
- データリーク防止のテストがある。

## 6. Phase 3: ルールベース予測・評価・分析

### 目的

Phase2の特徴量を使って、勝率・複勝率ではない相対的な予測スコアを生成し、過去結果に対する評価と分析を行う。

### 実装済み機能

- `prediction_runs`
- `race_predictions`
- `prediction_evaluations`
- ルールベース予測スコア生成CLI
- `rule-based-v1`
- `rule-based-v1.1`
- `--model-version` 指定
- 予測履歴画面
- 予測詳細画面
- 予測評価CLI
- 予測分析画面
- model_version別比較

### 重要条件

- 予測スコアは勝率・複勝率ではない。
- 買い目提案は行わない。
- 的中保証・利益保証をしない。
- 予測生成時に `race_results` を使わない。
- 評価時のみ `race_results` を検証ラベルとして使う。

### 完了条件

- 予測スコアを生成・保存できる。
- 予測根拠を画面で確認できる。
- 評価結果を保存できる。
- 分析画面でmodel_version別の傾向を比較できる。

## 7. Phase 4: AI/機械学習予測

### 目的

検証可能な機械学習モデルを設計・実装し、ルールベース予測と比較できるようにする。現時点では未実装。

### 候補機能

- モデル学習と評価
- 予測のバッチ生成
- モデルバージョン管理
- 予測根拠の説明
- 予測履歴と実績評価
- モデル監視

### 原則

- 最初はバッチ推論を優先し、リアルタイム推論は必要性を検証してから導入する。
- 評価指標、評価期間、控除率などの前提を明記する。
- 過学習、選択バイアス、データリークを継続的に検査する。
- 的中や利益を保証しない。
- モデルの不確実性と適用限界を表示する。

### Phase 4以降に検討するもの

- ペース予想
- 位置取り予想
- 複数モデルの比較
- 個人向けカスタマイズ

これらは独立した検証テーマとして扱い、一括して実装しない。

## 8. Phase 5: AWS移行・大規模化

### 移行を検討する条件

- SupabaseまたはVercelの費用が継続的に想定を上回る。
- ネットワーク、監査、可用性についてより細かな制御が必要になる。
- Pythonバッチや推論サービスがAWS中心になる。
- 組織・法人向けのセキュリティ要件が生じる。
- 単一クラウドに統合する運用上の利点が明確になる。

### 移行候補

- Next.js: VercelからAWS Amplifyへ
- PostgreSQL: Supabase PostgreSQLからRDS PostgreSQLへ
- ファイル: Supabase StorageからS3へ
- 認証: 当面Supabase Authを維持、または別途移行計画を策定
- バッチ: ECS Scheduled Tasks
- シークレット: AWS Secrets Manager
- 監視: CloudWatch

### 移行前の確認

- PostgreSQL拡張、関数、トリガー、RLSの互換性
- Supabase Auth IDとアプリケーションユーザーの対応
- データ量、停止許容時間、差分同期方法
- DB接続方式とコネクションプール
- ロールバック手順
- リハーサル結果

## 9. フェーズ変更管理

新機能を追加するときは、以下をIssueまたは設計文書に記録する。

- 所属フェーズ
- 解決するユーザー課題
- 使用するデータと利用条件
- セキュリティ・プライバシーへの影響
- データ時刻とリークへの影響
- ランニングコスト
- PostgreSQL移植性への影響
- 完了条件

フェーズ境界を越える変更は、単に実装可能という理由では着手しない。
