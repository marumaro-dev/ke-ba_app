# Codex開発ルール

> この文書は開発時の安全ルールです。フェーズの現在地と実装済み機能は [PROJECT_STATUS.md](./PROJECT_STATUS.md) を参照してください。

## 1. 文書の目的

本書は、Codexが本リポジトリで調査・設計・実装・レビューを行う際のプロジェクト固有ルールを定義する。

ユーザーから明示された指示が本書と矛盾する場合は、作業を止めて矛盾点を報告し、勝手に要件を変更しない。

## 2. 最優先原則

1. スクレイピングを実装・提案しない。
2. 契約・利用許諾が確認できないデータを取り込まない。
3. 的中保証、利益保証、回収率保証に当たる表現を作らない。
4. Phase 1範囲の作業ではAI予測、特徴量生成、ペース予想、位置取り予想を実装しない。
5. `available_at`、`observed_at`、`imported_at`を混同しない。
6. 秘密情報と個人情報をコード、ログ、テストデータへ含めない。
7. Supabase PostgreSQLからRDS PostgreSQLへの移行性を損なう変更を無断で行わない。
8. 依頼範囲外の大規模な変更や外部サービスへの書込みを行わない。

## 3. 作業開始時

Codexは変更前に以下を確認する。

- [PROJECT_OVERVIEW.md](./PROJECT_OVERVIEW.md)
- [PHASE_PLAN.md](./PHASE_PLAN.md)
- [ARCHITECTURE.md](./ARCHITECTURE.md)
- [DATABASE_DESIGN.md](./DATABASE_DESIGN.md)
- 現在のGit差分
- 対象機能に関係する既存コードとテスト
- データ提供元の仕様・利用条件がリポジトリ内にある場合はその文書

作業ツリーにユーザーの未コミット変更がある場合、それを消去・上書きしない。関係する変更と競合する場合は報告する。

## 4. スコープ管理

- Issueまたは依頼に必要な最小範囲を変更する。
- リファクタリングを同時に行う場合は、必要性を説明する。
- Phaseをまたぐ機能を「将来使うかもしれない」という理由だけで先行実装しない。
- 不明な要件を、競馬知識だけで推測して確定しない。
- データ提供元の仕様差を中核ドメインへ直接漏らさない。
- コード実装を依頼されていない場合は、コードや設定を作成しない。

## 5. データ取得ルール

### 許可されるもの

- 契約済みJRA-VANデータ
- 契約済みJRDBデータ
- 利用許諾と出所が明確なCSV
- 公式または正規に契約したAPI・データフィード
- テスト用に作成した合成データ

### 禁止されるもの

- Webページのスクレイピング
- robots.txtの可否だけを根拠にした収集
- 非公開APIの解析・利用
- 認証、レート制限、アクセス制御の回避
- 第三者が無断転載したデータの取込
- 契約上許可されない保存、加工、再配布

データ取得方法が不明確な場合、実装せず確認事項として報告する。

## 6. 時点データとリーク対策

外部データを保存する設計・実装では、必ず次を検討する。

- `available_at`: 情報が利用可能になった時刻
- `observed_at`: 値を観測した時刻
- `imported_at`: DBへ取り込んだ時刻
- `source_updated_at`: 提供元が示す更新時刻
- 速報、確定、訂正、取消などの状態

### 禁止事項

- `created_at`を無条件に`available_at`として扱う。
- 最新値で過去状態を上書きし、時点再現に必要な履歴を失う。
- レース結果や確定後情報を発走前データとして扱う。
- 訂正後の情報を訂正前から既知だったものとして扱う。
- 時刻不明の値を説明なく推定する。

時点の意味が不明な入力項目は、保存前に仕様確認を行うか、明確に隔離する。

## 7. DB設計・Drizzleルール

- DB変更はDrizzleスキーマとマイグレーションで管理する。
- Supabase Dashboardでの手動スキーマ変更を手順として提案しない。
- 外部IDを業務テーブルの主キーにしない。
- 外部キー、一意制約、CHECK制約を適切に使用する。
- 金額は整数、オッズなど精度が必要な値は`numeric`を優先する。
- 日時は原則`timestamp with time zone`、開催日は`date`を使用する。
- JSONBは原データの限定保存や可変メタデータに使用し、正規化すべき主要項目を押し込まない。
- N+1クエリを避ける。
- DB行をそのまま外部APIレスポンスにしない。
- 破壊的マイグレーションは段階的に行う。
- 適用済みマイグレーションを書き換えない。

### Supabase依存

- `auth.users.id`は`app_users.external_auth_id`を介して利用する。
- 業務テーブルの主キーにSupabase Auth IDを使用しない。
- RLSは明示的にテストする。
- Supabase固有拡張、関数、Realtime、Edge Functionsを追加する場合は、理由とRDS移行方法を文書化する。

## 8. TypeScript・Next.jsルール

- TypeScriptのstrict設定を維持する。
- `any`の追加を避け、必要な場合は理由をコメントまたは説明に残す。
- 外部入力はすべて信頼せず、Zodで境界検証する。
- Client Componentを必要最小限にする。
- 秘密情報を参照する処理はサーバー側だけに置く。
- 認証と認可を分けて考える。
- 認可をUIの表示・非表示だけで済ませない。
- Server ActionやRoute Handlerでも入力検証と権限確認を行う。
- ドメインルールをReactコンポーネントへ埋め込まない。
- 提供元固有の変換はアダプターに閉じ込める。
- エラーを握りつぶさず、安全な形で記録・通知する。

## 9. Zodルール

- フォーム、URLパラメータ、CSV、外部API、環境変数を検証する。
- DB型とAPI型が偶然一致することに依存しない。
- 文字列長、数値範囲、日付形式、列挙値を可能な範囲で制約する。
- エラーメッセージへ秘密情報や原データ全体を含めない。
- CSVなど大量入力では、行単位のエラー位置を追跡できるようにする。

## 10. 認証・セキュリティ

- SupabaseのAnon KeyとService Role Keyを区別する。
- Service Role Keyを`NEXT_PUBLIC_`環境変数へ設定しない。
- クライアントから送られたユーザーIDを所有者判定に使用せず、検証済みセッションから取得する。
- Cookie、セッション、CSRF、XSS、SQL Injection、ファイルアップロードを考慮する。
- アップロードファイルのサイズ、拡張子、MIME、内容を検証する。
- 管理用処理に一般ユーザーが到達できないことをテストする。
- ログにトークン、接続文字列、個人情報、契約上秘匿すべきデータを出さない。
- 依存パッケージ追加時は保守状況、ライセンス、脆弱性、バンドル影響を確認する。

## 11. 表現・UIルール

- 「必ず当たる」「利益が出る」「儲かる」などの保証表現を使用しない。
- 事実データ、ユーザーメモ、将来の予測データを視覚的・意味的に区別する。
- 速報、確定、訂正、更新時刻、出典を必要に応じて表示する。
- オッズや払戻の表示は、データ提供条件と更新時点を明示する。
- 将来の予測値を事実のように表示しない。
- アクセシビリティを考慮し、色だけで状態を区別しない。

## 12. テストルール

変更内容に応じ、以下を追加・更新する。

- ドメインルールの単体テスト
- Zodスキーマの正常・異常系テスト
- DB制約・リポジトリ層の統合テスト
- 認証・認可・RLSテスト
- 取込の冪等性テスト
- 訂正・取消・速報から確定への状態遷移テスト
- タイムゾーン境界テスト
- `available_at`基準の時点整合性テスト
- 主要画面の動作テスト

### 必須の代表ケース

- 同一ファイルを2回取り込んでも重複しない。
- 一部不正な行がある場合に、方針どおり全体失敗または部分成功になる。
- 他ユーザーのメモを閲覧・更新できない。
- 未認証・権限不足の管理処理が拒否される。
- 結果確定後の情報が発走前時点の抽出に含まれない。

## 13. CIルール

PR相当の変更では、少なくとも以下が成功する状態にする。

- Format
- Lint
- Type check
- Test
- Migration validation
- Build

テストを無効化してCIを通してはならない。既知の失敗が作業と無関係な場合は、変更せずに具体的な失敗内容を報告する。

## 14. ファイル・命名ルール

- 既存のプロジェクト構成が決まった後はそれに従う。
- 名前は英語で、役割が分かる語を使用する。
- 提供元名を中核ドメインの型名へ不要に含めない。
- `utils`、`helpers`へ業務ロジックを無秩序に集約しない。
- 1ファイルに複数の無関係な責務を持たせない。
- コメントは「何をしているか」より「なぜ必要か」を優先する。
- 文書とコードが矛盾する変更では、同じ変更内で文書も更新する。

## 15. 禁止コマンド・危険操作

明示的な許可なく以下を行わない。

- 本番DBへの接続・変更
- 本番マイグレーション
- 本番データのダウンロード
- `git reset --hard`など未コミット変更を失う操作
- 強制push
- ブランチやタグの削除
- 外部サービスへのデプロイ
- 課金サービスや有料リソースの作成
- ユーザー、Issue、PR、通知など外部状態の変更

## 16. ドキュメント更新条件

次の場合は該当文書を更新する。

| 変更 | 更新先 |
| --- | --- |
| 対象範囲、禁止事項、成功条件 | `PROJECT_OVERVIEW.md` |
| フェーズ、優先順位、完了条件 | `PHASE_PLAN.md` |
| サービス、責務、データフロー、認証方式 | `ARCHITECTURE.md` |
| テーブル、制約、時刻、移行方針 | `DATABASE_DESIGN.md` |
| 開発手順、品質基準、Codexの禁止事項 | `CODEX_RULES.md` |

## 17. 作業完了時の報告

Codexは次を簡潔に報告する。

- 変更したファイル
- 実装・設計した内容
- 実行した検証と結果
- 実行できなかった検証
- 残るリスクや未決定事項
- 推奨する次の作業

不確実な結果を成功と表現しない。コードを変更していない場合は、そのことも明示する。
